# v0.2.0 Architecture Review

**Subject:** `taskforge-skills` at tag `v0.2.0` (commit `8414ebd`)
**Date:** 2026-07-22
**Type:** pre-adoption architecture review — reviewing the released package as a
mature open-source project, not an implementation pass.

This document is a **reference record**, kept alongside `DESIGN.md` and
`taskforge/CONTRACTS.md`. It captures not only what to change but — in the
**Invariants to protect** and **Rejected** sections — what to deliberately
*keep minimal*, so future contributors do not "improve" load-bearing
decisions into complexity.

## How this review was produced

Six independent perspectives read the released code (not from memory):
Public API & Stability, Extension Model, Engine Architecture, Skill Design,
Developer Experience, Release & Maintenance. Each classified every finding as
`KEEP` / `IMPROVE-BEFORE-V1.0` / `FUTURE` / `REJECT` with a `file:line`
citation. The two **correctness** findings and the factual doc claims were
then verified by hand against the source. All sixteen sharp findings held —
a high hit rate that reflects a genuinely well-built package: the findings
are refinements, not structural holes.

The perspectives **converged** rather than disagreed. The one notable
convergence is called out below (schema evolution, flagged independently by
three perspectives) — that agreement is itself the signal to act.

## Verdict

**v0.2.0 is a solid architectural foundation for v1.0.** All six dimensions
returned "solid foundation." The core decisions — one deterministic engine as
sole writer, derived (not assigned) readiness, capabilities-as-registry, the
CONTRACTS-mandatory / references-on-demand split, and the doc-contract drift
net — are correct, internally consistent, and worth actively protecting.
Nothing is structurally unfinished.

The distance to a credible v1.0 is **two latent correctness bugs and one
undeclared contract**, not missing features. The package's problem is not
that it is too small; it is that a few edges of the small thing are not yet
nailed down.

## Tier 1 — Correctness (must fix before v1.0)

The only two blocking findings. Both verified by reading the code directly.

### T1-1. `apply_result` silently drops declared work on a circuit-breaker park, yet marks the result fully applied

**Evidence:** `taskforge/scripts/engine/apply.py:32-47`. When an artifact
trips the version breaker (`apply.py:108`) or review budget (`apply.py:97`),
the artifact loop breaks and the task parks (`blocked_on_human`);
`generated_tasks`, `edges`, and `signal` are then skipped by the
`if task["status"] != "blocked_on_human":` guard — but `result_id` is
appended **unconditionally** (`apply.py:47`) and the call returns
`applied: True`. A retry no-ops on the idempotency check.

**Why it matters:** realistic trigger — `taskforge-run` records an
out-of-scope follow-up task in the same result as a review that exhausts the
retry budget. The follow-up is lost, unrecoverable, and invisible. This is
silent loss of durable work in a system whose entire premise is durability.
**The single most important fix in the review.**

**Proposed direction (not prescribed):** apply `generated_tasks`/`edges`
regardless of the park — they do not depend on artifact convergence — while
keeping `signal` guarded; or withhold `result_id` on a mid-artifact park so a
`human-update`-driven re-apply can complete the dropped work. Either way, add
a test with a breaker-tripping artifact **plus** a follow-up in one result.

### T1-2. Stale-lock break is a TOCTOU race

**Evidence:** `taskforge/scripts/engine/store.py:133-137`. On
`FileExistsError`, a session reads the timestamp and, if older than
`LOCK_STALE_SECONDS` (60), `unlink`s the lock unconditionally and retries
`O_EXCL`. Two sessions crossing at the stale boundary can both unlink — and
B's unlink can delete A's *freshly acquired* lock, so both proceed holding it.

**Why it matters:** this breaks exactly the guarantee DESIGN §3.2.5 makes
("two sessions can't interleave a cascade"), in exactly the crash-recovery
path the timeout exists for. Low-probability, but a soundness hole in the
concurrency primitive that the whole single-writer guarantee rests on.

**Proposed direction:** make the steal atomic — `os.rename` a
uniquely-named temp lock into place instead of unlink+retry, so the loser's
rename fails and it re-reads. Do not widen the lock's scope.

## Tier 2 — The stability contract (what v1.0's number promises)

### T2-3. Declare the public boundary, and pin + test the output shapes

Two linked problems (api-stability findings #1 and #2):

- **Undeclared boundary.** `tasks.py:2-6` calls the Python facade "the
  engine's public interface," and `tasks.py:29-45` re-exports ~40 symbols
  including state-corrupting mutators (`supersede`, `cascade`, `materialize`,
  `block_on_human`). For an external consumer the real contract is the
  **CLI** — subcommand names, JSON output shapes, exit codes, actor names.
  The facade is an in-repo test convenience. Leaving both undelineated means
  v1.0 implicitly promises 40 function signatures and hands out
  Rule-Zero-violating mutators as "API."
- **Untested wire format.** The engine↔skill wire format *is* the JSON output
  field names — `references/reviewer-prompt.md:14` parses
  `next_review_version` (a field DESIGN.md:391-396 records adding
  specifically because a skill needed it); skills route on `readiness` and
  consume `blocking_ids`/`cycle` (`readiness.py:11-32`). These live only in
  code; `test_docs_contract.py` guards command *names* but not output
  *shapes*. A well-intentioned field rename silently breaks reviewer
  isolation and routing with no red build.

**Why it matters:** this is what makes the v1.0 version number honest. The
stable contract is currently implicit and partly untested.

**Proposed direction:** one paragraph (README/DESIGN) declaring the CLI —
subcommand names, documented JSON output shapes, exit codes, `--actor` names
— as the semver-governed public contract, and the Python facade as internal.
Pin the read-command output schemas (`readiness`, `budget`, `list`,
`show`/`summary`, `apply_result` return) in CONTRACTS.md or a short "engine
output" reference, and add a key-presence test — the same both-ways
discipline the command-table guard already applies to names.

**Sub-finding — `validate` returns a structurally-constant `valid: true`.**
`cli.py:168-173` always prints `valid: true` on the success path;
`validate_result` raises on any real violation → stderr + exit 1. The boolean
carries no information. Before freezing the shape at v1.0, either drop the
dead key (exit code + `warnings[]` is the contract) or make it meaningful.

### T2-4. Harden the schema-evolution story — flagged independently by three perspectives

The review's clearest convergence: api-stability, release, and
release-future all flagged the same subsystem — the one mechanism built
explicitly for "lives for years" is the least-exercised part of the engine.

- **Scan path bypasses the future-version guard.** `store.load()` raises when
  a task's `schema_version` exceeds the engine's (`store.py:83-86`) — correct
  fail-closed on the mutate path. But `all_tasks()` reads raw JSON with no
  check (`store.py:116-118`), and `list`/`blocked-by`/`doctor`/`migrate`/
  `refresh_dependents` all iterate it. In the git-tracked-store scenario the
  design explicitly supports (DESIGN §10.10), an older install silently
  computes wrong readiness for newer-schema tasks on the scan path.
- **`schema_version` + `migrate` ship untested.** Verified: zero tests touch
  `migrate` or `schema_version` — contradicting CONTRIBUTING's own rule ("New
  engine behaviour has a test … the engine is the thing prompts are allowed
  to trust").

**Why it matters:** the longevity thesis (DESIGN.md:352, README.md:106) rests
on data forward-compatibility, and that exact machinery is both half-applied
and unexercised.

**Proposed direction:** keep the mechanism (do **not** make `all_tasks`
hard-fail — that would break `doctor` on a mixed store). Have `doctor()`
surface any task whose `schema_version > SCHEMA_VERSION` as an explicit
finding, so shared-store drift is visible not silent. Add two engine tests
now, while the behavior is trivial: (1) a `schema_version=2` task makes
`load` raise; (2) `migrate` bumps an older task.

**Related (FUTURE, not blocking):** `migrate()` calls `store.save` without
`record()` (`audit.py:124`) — harmless while it is a no-op, but a real v2
migration would rewrite task fields invisibly to the event log. Note in
DESIGN §9 that any non-no-op migration must append a `migrated` event, so the
requirement lands with the first real migration.

## Tier 3 — Cheap correctness/consistency (batch before v1.0)

- **Version marker.** Frontmatter carries only name/description/license
  (verified across all four skills). An installed copy cannot say what version
  it is, contradicting the years-long thesis and the reason `license:` was
  added to frontmatter. Add `version:`, single-sourced, validator-checked
  against the CHANGELOG head.
- **Enum-surface drift guard.** `test_docs_contract.py` rigorously guards the
  command surface but nothing guards `SIGNALS`/`KINDS`/`RELATIONS`/actors —
  precisely the *harder* extensions (multi-site engine changes). A ~15-line
  stdlib guard mirroring `cli_subcommands()`, asserting each enum member is
  named in CONTRACTS, closes the asymmetry and fits the existing philosophy.

## Tier 4 — Documentation coherence (batch, doc-only, before v1.0)

- **HANDOFF.md** is a frozen snapshot presenting as live docs and has already
  drifted (missing `reopen`; missing `intake.md` in its §8 repo map). Add a
  dated "point-in-time, current architecture lives in DESIGN/CONTRACTS/
  CHANGELOG" banner, or move it to `docs/history/`.
- **README command list** (README.md:20-21) omits `budget` and `reopen`
  (verified). Fix it, and either replace the prose list with a pointer to the
  hub SKILL table or extend the doc-contract guard to cover it.
- **reopen/terminal-state and readiness routing are duplicated** across
  CONTRACTS.md:76-88 and `taskforge/SKILL.md:73-90` — violates the project's
  own single-source rule. Trim the SKILL to the command + operational tail;
  keep the concept in CONTRACTS.
- **DESIGN §9 oversells** "add a skill in one file" — true only for
  adjunct/query skills; a new *routed* workflow stage needs a readiness rule +
  `STATUS_FOR` entry + capabilities. Amend to distinguish the two skill
  classes.
- **readiness `human`** appears in the CONTRACTS table only as the
  parenthetical `(cycle → human) → waiting` (CONTRACTS.md:131); `evaluate()`
  emits it as a distinct value and `list --readiness human` accepts it. List
  it explicitly as a readiness value with its meaning.
- **Link the walkthrough.** `examples/walkthrough-m3.md` (the only end-to-end
  narrative) is referenced only by the frozen HANDOFF. Link it from README and
  rename it off the internal `m3` milestone tag.

## Invariants to protect (KEEP — do not "simplify" these away)

These are correct *because* they are minimal. Each is one refactor away from
being broken by a well-meaning contributor; they are recorded here so that
does not happen silently.

1. **Dual status/readiness vocabulary.** Cached display strings
   (`ready_to_run`, `needs_refine`, …; `model.py:27`) are subordinate to
   derived readiness; consumers are told to route on readiness only
   (CONTRACTS.md:69). This is the derived-not-assigned principle made
   concrete. Do not unify the two vocabularies.
2. **The bimodal extension model.** An open data plane (actors, annotation
   edges, templates, references — zero engine edits) around a deliberately
   closed semantic core (kinds/signals/relations carry cascade/readiness
   semantics and must not be configurable). Do not promote annotation edges or
   actors into engine constants; do not make the semantic core a plugin
   surface.
3. **Module DAG + lock-at-boundary.** The acyclic dependency graph DESIGN §3
   claims actually holds (model → store → readiness/validation → apply →
   audit → cli, no cycles); `audit` is deliberately off the write path. Add an
   import-graph CI assertion so a back-edge cannot sneak in.
4. **Determinism.** `sort_keys` on write (`store.py:104`), `sorted(glob)` on
   scan (`store.py:117`), `max(version)` selection (`model.py`); no
   clock/RNG/dict-order dependence in state or routing.
5. **Idempotency ordering.** Dup-check *before* validate (`apply.py:20-28`),
   so a retry of a result that already made the task terminal no-ops cleanly.
   Subtle, correct, tested, and a settled §10 decision.
6. **Boundaries enforced by `capabilities.json`, not just prose.** Each actor
   owns exactly one artifact domain. Do not add a second artifact kind to any
   actor, and do not let the hub skill "just run" the next skill.
7. **Dual-trigger descriptions + mandatory readiness guard.** Description
   overlap is self-correcting because the readiness guard bounces a
   mis-selected skill before any mutation. Do not "fix" the overlap.
8. **Explain-the-mechanism-never-enforce-it.** Skills query the engine for
   every deterministic quantity. The moment a prompt computes budget/version/
   readiness math the model can act on, it becomes a second source of truth.
9. **The doc-contract suite itself.** The mechanism that keeps skill docs
   honest against the engine, retrospective and stdlib-only. Do not collapse
   the two test roots.

## Rejected (a plausible change that would harm the design)

**Do not abstract the enum dispatch into a plugin/registry.** The parallel
`if/elif` chains over the model enums (`validate_payload` per KIND,
`apply.py:192` `apply_signal` per SIGNAL, `materialize` per RELATION) *look*
like calcification you would "fix" with a registry. Rejecting that is right:
the vocabularies are tiny and fixed by design (KINDS=4, SIGNALS=6,
RELATIONS=3), each branch's behavior is genuinely distinct, and a registry
would add indirection to save nothing while inviting exactly the
"configurable semantic core" the architecture correctly refuses. The friction
of a multi-site enum change is proportional and intentional; the enum-surface
drift guard (Tier 3) makes that friction *safe* rather than abstracting it
away.

## Highest-leverage investments before v1.0

If only three things are done:

1. **Fix the partial-apply data loss (T1-1)** — the one genuine latent bug,
   and it is silent loss in a durability-first system.
2. **Declare + test the output-shape contract (T2-3)** — this turns "v1.0"
   from a number into a promise; without it the stability story rests on field
   names no test defends.
3. **Harden schema evolution (T2-4)** — three perspectives independently
   flagged the same subsystem as the weakest part of the longevity thesis.

The lock race (T1-2) is cheap; fold it into the same correctness pass.
Everything else is batchable doc/consistency cleanup or safely deferrable.

## On "unfinished" vs. "intentionally minimal"

The line is drawn correctly almost everywhere. The genuinely *unfinished*
items are narrow: two correctness gaps, the untested migration path, the
undeclared output contract. Everything that looks sparse — the ubuntu-only CI
matrix, the four-value routing vocabulary, the concrete enum dispatch, the
absence of a plugin loader — is intentionally minimal and should stay.
